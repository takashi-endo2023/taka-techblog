## 「設計に詰まる」瞬間

新機能の実装前、設計に悩む瞬間は誰にでもあります。

「このバリデーションはサービス層に置くか、ドメイン層に置くか」「このキャッシュは Redis か、インメモリか」「マイクロサービス化のタイミングはいつか」——こういった問いに対して、即座に答えを持てるわけではない。

チームに相談できる相手がいれば良いですが、少数精鋭チームでは技術的な深い相談ができる人間が限られています。そこで Claude Code を「設計の壁打ち相手」として活用するようになりました。

## コード生成ツールではなく「思考の整理ツール」として

Claude Code の強みは、コードを書くことだけではありません。コンテキストをたっぷり与えて「どう設計するか」を一緒に考える相手として使うことで、一人で詰まっていた問題が30分で解決することがよくあります。

### 実際のプロンプト例

```
NestJSのCRMシステムで、以下の設計判断で迷っています：

【背景】
- チーム5名（うちエンジニア3名）の医療系スタートアップ
- CRMはNestJS + TypeORM + PostgreSQL
- 月次のバッチ処理でCSVを取り込み、顧客データを更新する機能を追加したい

【選択肢A】CSVバッチ処理をNestJSのScheduleモジュールで実装
- pros: 既存のNestJSエコシステムに統合できる
- cons: APIサーバーとバッチが同プロセスになる

【選択肢B】AWS Lambda + S3イベントトリガーで別サービスとして実装
- pros: APIサーバーへの影響ゼロ、スケーラビリティ
- cons: インフラ管理が増える、NestJSとTypeORMの共有が難しい

【選択肢C】BullMQ（Redisキュー）でジョブとして非同期処理
- pros: 進捗管理・リトライが簡単、APIサーバーと分離可能
- cons: Redisが増える

この3つのトレードオフを整理して、チームの状況を踏まえておすすめを教えてください。
```

このように「背景・選択肢・pros/cons」を事前に整理してから投げると、出力の質が上がります。

## Claude Codeが返してくること

上記のようなプロンプトに対して、Claude Codeは：

1. 各選択肢のトレードオフを再整理
2. チームの状況（5名・医療系・少数精鋭）を考慮した推薦
3. 見落としていた観点の提示
4. 選んだ場合の実装の始め方

を返してきます。

「正解を教えてもらう」というよりも「自分の思考が整理される」感覚に近い。プロンプトを書く過程で「あ、自分はAとBで迷っているが、実はCという選択肢を考えていなかった」と気づくことも多い。

## 壁打ちを効果的にするコツ

### 1. 前提をたっぷり書く

技術スタック・チームの規模・スキルセット・制約条件をできるだけ詳しく書きます。コンテキストが多いほど、回答の精度が上がります。

### 2. 自分の仮説を先に書く

「選択肢A・B・Cがある」だけでなく「自分はAが良いと思っているが、XXという懸念がある」と書くことで、懸念点に対してピンポイントで応答してもらえます。

### 3. 「なぜ」を聞く

推奨をもらったら「なぜAではなくBなのか、Aの場合にどんな問題が起きる可能性があるか」をフォローアップで聞きます。設計の背景を理解することで、将来の判断に活きます。

### 4. CLAUDE.mdにプロジェクトコンテキストを書いておく

```markdown
# プロジェクト概要
- 医療系スタートアップのCRMシステム
- NestJS + TypeORM + PostgreSQL
- チーム: エンジニア3名（うち2名は未経験から育成中）
- 制約: 規制医療情報を扱うため、外部サービスへのデータ送信に注意

# 設計方針
- シンプルさを優先（複雑なアーキテクチャは採用しない）
- TypeScriptの型を活用
- テスト容易性を意識した設計
```

CLAUDE.mdを充実させることで、毎回同じ前提を書かなくて済みます。

## 実際に壁打ちして決めた設計判断

上記の CSV バッチ処理の相談では、最終的に「BullMQ + Redis」のオプションCを選びました。

理由：
- CSVインポートは時間がかかる可能性があり、APIサーバーをブロックしたくない
- 進捗状況をフロントエンドに表示する要件が後から出てくることが予想された
- Redisは既に別用途（セッション管理）で使っており、新規追加コストが低い

この判断を30分の壁打ちで整理できたのは大きな価値でした。一人で考え続けると半日〜1日かけることもある。

## 「思考の劣化」に注意

一点、使い続けて感じたリスクがあります。

壁打ちが便利すぎると「自分で考えること」をサボるようになるリスクがあります。「Claude Codeに聞けばいい」という癖がつくと、設計力そのものが育たなくなる。

私は「まず10〜15分は自分で考える」「自分の仮説を持ってから相談する」というルールを設けています。思考補助ツールであって、思考代替ツールではない、という意識が大切です。

## まとめ

- Claude Codeはコード生成だけでなく設計相談にも使える
- 背景・選択肢・自分の仮説を事前に整理して渡す
- 出力は「答え」ではなく「思考の整理」として活用
- CLAUDE.mdにプロジェクトコンテキストを書いておくと効率的
- 「自分で考える習慣」とのバランスを意識する
