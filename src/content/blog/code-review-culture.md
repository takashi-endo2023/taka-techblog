コードレビューが「形式的な承認作業」になっているチームがある。PRが出ると、細かい指摘も質問もなく、数分後に「LGTM 👍」がついてマージされる。

これは「レビューをやっている」のではなく「レビューの儀式をやっている」だ。

私がジョインしたチームがまさにそれだった。3人のエンジニアがいて、PRが出るたびに素早くLGTMがつく。品質チェックも知識共有も何も起きていなかった。

テックリードとしてコードレビュー文化を作り直した記録を書く。

## なぜLGTMレビューになっていたのか

まず「なぜそうなっているか」を理解することから始めた。責める前に理解する。

1on1で聞いてみると、理由は明確だった。

**「何を見ればいいかわからない」**

「コードが動いていればOK」という基準しかなく、それ以外に何を確認すべきかが曖昧だった。

**「指摘すると人間関係が悪くなりそうで怖い」**

チームに「フィードバックは攻撃」という暗黙の認識があった。指摘されると傷つく人がいて、だから誰も指摘しなくなっていた。

**「レビューより実装の方が楽しい」**

これは正直な意見だ。コードを書く方が達成感があり、レビューは後回しになりやすい。

この3つの問題をそれぞれ解消することが必要だった。

## やったこと1：「何を見るか」を明文化した

まず「コードレビューチェックリスト」を作った。

```markdown
## 機能面
- [ ] 仕様通りに動作するか
- [ ] エラーケースが考慮されているか（null・空配列・ネットワークエラーなど）
- [ ] セキュリティ上の問題がないか（入力値の検証・権限チェック）

## 設計面
- [ ] 変数名・関数名が意図を表しているか
- [ ] 関数が長すぎないか（目安: 30行以内）
- [ ] 同じコードが繰り返されていないか（DRY）

## テスト面
- [ ] 新機能にテストが追加されているか
- [ ] バグ修正の場合、再現テストが追加されているか

## ドキュメント面
- [ ] 複雑なロジックにコメントがあるか
- [ ] 外部から見えるAPIにJSDocがあるか
```

チェックリストの効果は大きく、「何を確認すればいいかわからない」問題がほぼ解消した。レビュアーが観点を持てると、コメントの質も上がった。

## やったこと2：コメントの文化を作った

レビューコメントの書き方を「提案型」に統一した。

```
# 指摘型（避ける）
これは間違っています。

# 質問型（良い）
この部分、null の場合はどうなりますか？

# 提案型（良い）
この処理、こう書くと読みやすくなるかもしれません：
```typescript
const result = items.filter(Boolean).map(transform);
```

# 称賛型（積極的に使う）
このエラーハンドリング、丁寧に書いてあってとても読みやすいです！
```

「指摘」ではなく「質問」と「提案」でコメントするルールを徹底した。「なぜそう書いたのか聞いてみる」というスタンスが、心理的安全性を高める。

また、良いコードへの「称賛コメント」を積極的に入れることも大切だ。レビューが「ネガティブなことを言う場」になると萎縮するが、「良いコードを褒める場」も兼ねていると、レビューを受けることへの抵抗が減る。

## やったこと3：レビュー期限を設けた

「PRが出たまま3日放置される」という問題があった。実装者は「早くマージしたい」のに、レビュアーは他の作業を優先してしまう。

ルールを作った：**PRが出たら24時間以内に最低1人がコメントを付ける**。

最初のコメントは「全部見た」でなくていい。「見始めました。この部分だけ先に確認します」でも価値がある。「誰かが見てくれている」という安心感が実装者に届く。

これを守るために、PRが24時間以上コメントなしの場合はSlackに自動通知するGitHub Actionsを作った。

```yaml
# .github/workflows/pr-reminder.yml
name: PR Review Reminder
on:
  schedule:
    - cron: '0 9 * * 1-5'  # 平日朝9時に実行

jobs:
  remind:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/github-script@v6
        with:
          script: |
            const prs = await github.rest.pulls.list({
              owner: context.repo.owner,
              repo: context.repo.repo,
              state: 'open'
            });
            // 24時間以上コメントなしのPRをSlackに通知
```

## やったこと4：ペアレビューで観点を共有した

週に1回、30分のペアレビューセッションを設けた。全員でひとつのPRを一緒に見る時間だ。

目的は「レビューの手間を減らす」ことではなく、「レビューの観点を共有する」ことだ。「ここを見てほしい」「自分はここが気になった」という会話をリアルタイムでするうちに、チーム全体のレビューの質が上がっていく。

最初は「週30分もったいない」という声があったが、3ヶ月後には「このセッションが一番勉強になる」と言う人が出てきた。

## 半年後の変化

コードレビュー文化を作り直してから半年後、数字で変化が出た。

- PRのLGTMまでの平均時間：3日 → 1日以内
- 1PRあたりの平均コメント数：0.3件 → 4.2件
- 本番でのデグレ件数：月3〜4件 → 月0〜1件

数字より大切だと感じた変化は「レビューコメントへの反応が変わった」ことだ。以前はコメントが来ると「また指摘された」という空気があったが、今は「ここで聞いてよかった」「気づかなかった、ありがとう」という反応が増えた。

コードレビューが「指摘する場」から「チームで学ぶ場」になったと感じている。

## テックリードの役割

コードレビュー文化を作る上でテックリードがやるべきことは「ルールを決める」より「自分が一番良いレビューをする」ことだと気づいた。

チームはリーダーを見ている。自分が丁寧な質問型・提案型のコメントをし続けることで、それが「このチームのレビューの作法」になっていく。言葉でルールを説くより、行動で文化を作る方が圧倒的に速い。
