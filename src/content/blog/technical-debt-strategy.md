「技術的負債を返済したい」

テックリードになってから何度この言葉を口にしたかわからない。でも「返済したい」と言うだけでは何も変わらない。技術的負債を実際に減らすには、戦略が必要だ。

医療系スタートアップで積み上がった負債に向き合った3年分の試行錯誤から、「これが機能した」というアプローチをまとめる。

## 技術的負債の「見える化」から始めた

最初の問題は「負債がどこにどれだけあるか」が誰にもわかっていないことだった。エンジニアは感覚的に「あそこは触りたくない」と知っているが、それを経営層やPMに説明できる形になっていなかった。

**やったこと: 負債マップの作成**

コードベースを機能領域ごとに分け、各領域の「負債スコア」を可視化した。スコアは以下の観点で決めた。

| 観点 | 内容 |
|---|---|
| 変更頻度 | よく変更される領域の負債は優先度が高い |
| バグ発生率 | 過去半年でバグが多く出た箇所 |
| テストカバレッジ | テストがない領域はリスクが高い |
| 理解のしやすさ | 新規メンバーが読んで理解できるか |
| 外部影響範囲 | 変更が他の多くの箇所に影響するか |

これを表にして、毎月チーム全員で更新する習慣にした。「何が問題か」が共通認識になると、議論の質が変わった。

## 「全部直す」をやめた

最初に犯したミスは「全部直そうとした」ことだ。

「このコードは全体的によくない。大規模リファクタリングをスプリントに組み込もう」という提案をしたとき、PMから「それはユーザーに何の価値も届かない。今は機能開発を優先してほしい」と言われた。

この言葉は正しかった。

技術的負債の返済は、エンジニアにとっては重要でも、ビジネスにとっては「今すぐ必要かどうか」という判断になる。「全部直す大規模リファクタリング」は、ビジネスから見ると「数週間機能が増えない状態」だ。

**考え方を変えた：ボーイスカウトルールの徹底**

「キャンプ場は来たときより美しくして去れ」というボーイスカウトルールをチームの文化にした。機能開発や不具合修正で触ったコードは、ついでに少しだけ改善する。

```
// PRのチェックリストに追加
- [ ] 今回触ったファイルで、気になる点を1つ改善した（コメント追加・命名改善・小さな抽出など）
```

大規模な一括修正より、小さな改善の積み重ねの方が継続できる。3ヶ月後にコードを見ると、確実に読みやすくなっていた。

## ステークホルダーへの説明方法

技術的負債をエンジニア以外に説明するのが苦手だった。「コードが汚い」「設計がよくない」という言葉は技術者にしか伝わらない。

**翻訳の方法：ビジネスへの影響で話す**

```
× 「この認証モジュールは凝集度が低く、単一責任の原則に違反しています」
○ 「この部分に変更が必要になるたびに平均2日の作業が発生していて、
   同規模の変更が他の領域では半日で完了しています」
```

「どこが汚いか」ではなく「どれだけ開発コストがかかっているか」で話すと、PMも経営層も理解しやすくなる。

特に効いたのが「過去の事例」だ。「このモジュールに触るたびに平均1.5件のデグレが発生している（過去3ヶ月のデータ）」という具体的な数字は、誰にでも伝わる。

**技術的負債の返済を機能開発と一緒に計画する**

「負債返済のスプリント」は機能開発が止まるため承認されにくい。代わりに「機能追加の際に一緒に改善する」という計画を立てた。

「ログイン機能を改善するタスクに、認証モジュールのリファクタリングをセットにする」という形だ。ユーザーへの価値（ログイン改善）と負債返済（リファクタリング）がセットになると、PMも合意しやすい。

## 「触れないコード」への対処

チームに必ず「誰も触りたがらないコード」が存在する。複雑すぎて怖い、あるいは書いた人しか理解できない、というやつだ。

治験システムにも、3000行のGodクラスがあった。「被験者管理」という名前で、被験者の登録・更新・検索・ステータス管理・権限チェック・ログ記録・通知送信が全部入っていた。

**段階的な「理解→テスト→分割」のサイクル**

1. **まず理解する**: ペアプロでコードを声に出して読む。「ここで何をしているか」を言語化する
2. **テストを書く**: 理解した振る舞いをテストとして記録する（リファクタリングの安全網）
3. **少しずつ抽出する**: 明らかに独立した機能を別クラスに抽出する

一気に分割しようとすると詰まる。まず「何をしているか理解する」だけでも価値があり、次のエンジニアが触りやすくなる。

## 負債を「返済し続ける仕組み」を作る

技術的負債は一度返済しても、また積み上がる。だから「返済する仕組み」を開発プロセスに組み込んだ。

**設計レビューをPRに組み込む**

新しい機能を実装するとき、コードを書く前に「このアプローチで技術的負債を生まないか」という設計議論をSlack上でする。非同期で相談できる小さな仕組みだが、「後から気づく問題」を「前に気づける問題」に変える効果がある。

**四半期の負債返済タスクの積み立て**

スプリントのベロシティの10%を技術的負債の返済に使う、というルールを設けた。「1スプリント3ポイント分は負債返済タスクをPickupする」と決めると、小さな改善が継続的に行われる。

最初は「なぜ機能開発に使えるポイントを削るのか」という声があったが、「負債が溜まると機能開発が遅くなる。事前投資だ」と説明して合意した。

## 3年間の変化

3年前と比較して、今のコードベースは別物だと感じる。

大規模リファクタリングは1度もしていない。ボーイスカウトルールの積み重ねと、機能開発と一緒に進める小さな改善の積み重ねだけだ。それでも「新しいメンバーがコードを読んで理解できる」「変更が怖くない」という状態になった。

技術的負債は「返済する問題」ではなく「管理し続ける問題」だということが、今では腑に落ちている。
