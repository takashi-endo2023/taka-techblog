「TypeScriptに移行したい」と思ってから、実際に移行が完了するまで8ヶ月かかった。

そのうち最初の3ヶ月は「どうやって移行するか」より「チームをどう納得させるか」に費やした。技術的な作業より、合意形成の方が圧倒的に時間がかかる。

移行完了後に振り返ると、「やってよかった」と全員が言っていた。でもプロセス中は何度も「本当に必要か」という声が出た。その泥臭い実録を書いておく。

## なぜ移行しようと思ったか

治験管理システムを開発する中で、JavaScriptの「型がない」ことによる問題が積み重なっていた。

一番痛かったのは、外部APIのレスポンス形式が変わったことに気づかずバグを出したことだ。APIのレスポンスが `{ status: "ok" }` から `{ result: "success" }` に変わったのに、コードは `.status` を参照し続けていた。テストがなければ本番リリースまで気づかない。

TypeScriptなら、APIのレスポンス型を定義しておけば、コンパイル時に「`result` プロパティは存在しない」とエラーが出る。型がバグを未然に防いでくれる。

もうひとつは「コードの読みやすさ」だ。引数の型がわからない関数は、使い方を理解するために定義元まで追いかける必要がある。TypeScriptなら型がドキュメントになる。

## チームへの説明で使ったロジック

「TypeScriptにしたい」とだけ言うと「学習コストがかかる」「移行に時間がかかる」という反対意見が出た。それは正しい指摘なので、「なぜやるか」をデータで示した。

**示したこと:**
- 過去3ヶ月で発生したバグのうち、型エラーで防げたと推定されるものの割合（約40%）
- TypeScript移行済みの同規模プロジェクトでのバグ件数比較（GitHubの公開事例）
- 移行コストの見積もり（段階的移行なので1スプリントに少しずつ）

「すぐに全部移行する必要はない。新規ファイルからTypeScriptで書くだけでいい」という提案が最終的に受け入れられた。ゼロイチの話ではなく、段階的に進めることを明示したのが効いた。

## 移行の進め方：段階的アプローチ

### Phase 1：設定だけ入れる（1〜2週間）

まず `tsconfig.json` を入れ、既存の `.js` ファイルは触らない。`allowJs: true` にすれば、JSとTSが混在できる。

```json
{
  "compilerOptions": {
    "target": "ES2020",
    "strict": false,
    "allowJs": true,
    "checkJs": false,
    "outDir": "./dist"
  }
}
```

この時点では何も壊れない。「TypeScriptが入っている」という事実だけ作る。

### Phase 2：新規ファイルはTypeScriptで（2〜3ヶ月）

新しく作るファイルは全て `.ts` で書く。既存のJSは触らない。

この期間で重要なのは「`strict: false` から始めること」だ。最初から厳格な型チェックをオンにすると、型エラーが大量に出て嫌になる。まず「型を書く習慣をつける」ことを目標にする。

### Phase 3：既存ファイルの段階的移行（3〜5ヶ月）

優先度の高いファイルから少しずつ `.js` → `.ts` に変換する。

優先度の判断基準は「変更頻度が高い」「バグが出やすい」「他から多く参照される」だ。コアのビジネスロジックや型安全性が特に重要な外部連携部分から始めた。

`// @ts-nocheck` を使うと特定のファイルを一時的に型チェックから除外できる。「全部移行するまで型チェックを無効にする」ではなく、「移行済みのファイルだけ型チェックをする」という使い方だ。

### Phase 4：`strict: true` に移行（移行完了後）

全ファイルがTSになったら、`strict: true` に上げる。これで `strictNullChecks` などが有効になり、型の恩恵を最大限受けられる。

ここでまた型エラーが大量に出るが、それは「今まで潜在していたバグ」の表面化だ。一つ一つ直すことでコードが堅牢になる。

## 移行して気づいた「型のある安心感」の正体

移行前は「TypeScriptは型を書く追加作業」だと思っていた。移行後に気づいたのは、型は「作業」ではなく「設計」だということだ。

型を書くことで、「この関数は何を受け取って何を返すか」を考える習慣がつく。引数が増えすぎた関数に型を書こうとすると「これは設計がおかしい」と気づける。型がコード設計の指標になる。

また「型があるから安心してリファクタリングできる」という体験を何度もした。リファクタリングして型エラーが出なければ、少なくとも型レベルでは壊していない。TypeScriptがない時代、大きなリファクタリングは「なんとなく怖い」ものだったが、今は「型が通っていれば問題ない」という判断軸が持てる。

## チームの反応

移行が完了した後のチームの反応は「最初から入れておけばよかった」だった。

学習コストは確かにあったが、「型がないときの不安感」が消えたことの方が大きかったという声が多かった。特に治験システムのような「バグが許されない」環境では、「コンパイル時に問題を発見できる」安心感がチームの精神的な負担を下げた。

移行を始めてから8ヶ月後、新しいメンバーが加入したとき「TypeScriptがあるから読みやすい」と言っていた。それが一番の報酬だった。
