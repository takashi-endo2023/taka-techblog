最初からブレていない目標がある。**私が暇になること。**

「暇になる」というのは怠けたいわけじゃない。今やっていることを誰かに渡せれば、次の新しいことを自分が先行して始められる。そして後に続いてくれると信じている。

## なぜ3年間、一人でやるしかなかったか

コードレビューは、ベンダーからシステムを引き継いだ日から今まで、自分がやってきた。

ただ「他に人がいなかった」だけではない。このプロダクトが持つ構造的な複雑さが、レビューを一人に集中させていた。

うちのシステムは、治験CRMと仕込みサイトが**同じアーキテクチャで連動している**。関数の共通化が進んでいるため、片方の変更が意図せずもう片方に影響することがある。影響範囲が見えにくい。「このPRは大丈夫か」の判断には、両方のシステムの文脈を頭に持っておく必要があった。

チームに未経験の2人がいる。育てながら一緒に動いてきたが、この文脈を持てるようになるには時間がかかる。全PRの最終判断を自分がやる以外になかった。

ボトルネックは自分だとわかっていた。でも他に選択肢がなかった。

## 手を離した瞬間

数ヶ月前から、先輩メンバーにコードレビューを任せ始めた。

正直、怖かった。

品質への不安もあるし、何かミスが起きたときの責任の話もある。でも「チームを前に進めるには段階的委譲をすべき」という判断は変わらなかった。リスクは承知した上でやる。覚悟の問題だと思った。

やり方は段階を踏んだ。最初は自分と二重でレビューしていた。彼女のレビューを確認してから、自分もレビューしてマージする。しばらくそれを続けた。

次のステップとして、**検証環境（stgブランチ）へのマージは彼女の承認だけで通せるようにした**。mainブランチへのマージは今でも自分がやっている。最後のリスク回避として、本番に近い段階だけは自分が見る。

結果——意外となんとかなっている。バグはゼロだ。

「任せてみないとわからないこともある」と気づいた。3年間ずっとボトルネックだった自分から、初めて一部が剥がれた。小さい変化に見えるかもしれないが、自分にとっては大きかった。

## 次は仕組みに渡す

人への委譲と並行して、仕組みへの委譲も進めている。個人のスキルに依存したレビューを、ルール化・平準化して自動化する。今取り組んでいることと、これから目指すことを段階で書く。

### ステージ1（進行中）：結合試験の GitLab CI 自動化

手動でやっていた結合試験を、CI上で自動化する。

設計はClaude と一緒に進めている。「人の手をかけない、誰がやっても同じ品質になる」というゴールを起点に、何が技術的に実現できるか、どの構成が現実的かを壁打ちしながらイメージを具体化している。

構成のイメージは、GitLab CIでstgブランチとmainブランチへのマージ時に結合試験が走り、通ればマージOKという形だ。stg環境でボタンを押せば試験が走るようにしてもいい。

現状は他部署からの最優先依頼に割り込まれていて、作業が止まっている。ただ、この自動化が終われば、将来の依頼への対応スピードも上がる。「待っていてくれ、これが先だ」と言うつもりでいる。自動化への投資を自分の言葉で説明できるようになってきた。

### ステージ2（次）：Claude Code によるコードレビュー

PRが出たタイミングで、AIが一次レビューを行う仕組みを作りたい。

具体的には、Claude Code に CLAUDE.md でレビュールールを定義して実行させることができるんじゃないかと考えている。まだ妄想の段階だが、実現できれば命名・重複・潜在バグ・セキュリティの観点を自動でチェックできる。人間のレビューは設計判断に集中できる。

特に**潜在バグの検知**に期待している。前述のとおり、治験CRMと仕込みサイトが連動しているため、関数の変更が思わぬ場所に波及するリスクがある。この文脈をCLAUDE.mdに書いておけば、AIがそのリスクを意識した観点でレビューしてくれる可能性がある。

### ステージ3：マージ段階での不具合検知

mainへのマージ前に自動で不具合を検知する仕組みを作る。品質ゲートを自動化する。ここまでできれば、自分がmainのマージ承認者でいる必要もなくなってくる。

### ステージ4：開発フロー全体の自動化

可能な限り、一連の開発フロー自体を自動で回せる状態にしたい。「自分がいなくても品質が保たれる」仕組みを作ることが、このロードマップの終着点だ。

## 「私がいなくても回る組織」がずっとゴール

内製化を進めたのも、未経験の2人を育ててきたのも、AIを使った自動化に取り組んでいるのも、全部つながっている。

どれも「自分がボトルネックであることをやめる」ための行動だ。

先輩への委譲が意外となんとかなったように、自動化も「やってみないとわからない」ことがたくさんある。完成形を描いてから動くより、動きながら形を作る方が自分には合っている。

暇になれば、次のことを始められる。自分が先頭に立って新しいことを開拓して、チームが後に続いてくれる。その繰り返しで、組織は育つと思っている。

今はまだ全然暇じゃない。でも方向性は変わらない。
