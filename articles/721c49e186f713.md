「とりあえずCache-Control: no-cache」で済ませているなら、もう少し深く理解する価値がある。

## よくある誤解

`no-cache`は「キャッシュしない」ではなく**「キャッシュするが毎回検証する」**。完全に無効にしたいなら`no-store`。

## Cache-Controlの主要ディレクティブ

```http
# 300秒間キャッシュ（ブラウザのみ）
Cache-Control: private, max-age=300

# CDN・プロキシも含めて1時間キャッシュ
Cache-Control: public, max-age=3600

# CDNには1日、ブラウザには1分
Cache-Control: public, max-age=60, s-maxage=86400

# キャッシュするが毎回検証（ETagと組み合わせる）
Cache-Control: no-cache

# 一切保存しない
Cache-Control: no-store
```

## ETagによる条件付きGET

```http
# 最初のレスポンス
HTTP/1.1 200 OK
ETag: "abc123"
Cache-Control: no-cache

# 次のリクエスト
GET /api/users/123
If-None-Match: "abc123"

# 変更なし → ボディなしで返す（通信量節約）
HTTP/1.1 304 Not Modified

# 変更あり → 新しいETagと本文を返す
HTTP/1.1 200 OK
ETag: "xyz789"
```

## CDNキャッシュ設計

```http
# 静的ファイル（ファイル名にハッシュあり）
Cache-Control: public, max-age=31536000, immutable

# HTML
Cache-Control: public, max-age=0, s-maxage=3600

# 公開API
Cache-Control: public, max-age=300, stale-while-revalidate=60

# 認証済みAPI
Cache-Control: private, max-age=60

# リアルタイムデータ
Cache-Control: no-store
```

## Next.js App Routerでの設定

```ts
// API Route
return Response.json(data, {
  headers: { 'Cache-Control': 'public, max-age=300, stale-while-revalidate=60' },
});

// Server ComponentでのFetch
const data = await fetch('/api/data', { next: { revalidate: 300 } });
const live = await fetch('/api/live', { cache: 'no-store' });
```

## まとめ

| シナリオ | 推奨設定 |
|---|---|

| HTML | `public, max-age=0, s-maxage=3600` |
| 公開API | `public, max-age=300, stale-while-revalidate=60` |
