あれは治験システムの被験者管理モジュールに機能追加をリリースした翌朝のことだった。

朝イチで「来院スケジュールの一覧が表示されない」という問い合わせがモニタリングチームから来た。ステージング環境では問題なく動いていたはずなのに——そこから数時間、原因を追うことになった。

## 何が起きたか

その週、2つのチケットが並行して走っていた。一方は来院スケジュール機能への絞り込み条件追加、もう一方は被験者ステータス変更ロジックのリファクタリングだ。どちらも同じファイルを触っていたため、マージ時にコンフリクトが発生した。

コンフリクト解消は自分が担当した。マーカーを消して、両方の変更を取り込んで、「よし、解消完了」とプッシュした。ステージングでは来院スケジュール一覧も表示され、ステータス変更も動いた。リリースは問題なく完了したと思っていた。

だが実際には、コンフリクト解消の際にリファクタリング側のコードの一部を誤って上書きしていた。具体的には、来院スケジュールを取得するクエリに**被験者の有効フラグによるフィルタリング条件**が含まれていたのだが、その行がコンフリクト解消の過程で消えていた。

ステージング環境には有効フラグが全部 `true` のテストデータしかなかったため、フィルタリングが機能しなくても結果は変わらなかった。本番には無効化された被験者レコードが存在しており、それらも一覧に表示されてしまったことで障害として発覚した。

## 発覚の経緯と初動対応

報告を受けてまず確認したのは、直近のデプロイタイミングと障害の発生タイミングだ。前夜のリリース直後から症状が出ていたことがログから確認でき、リリース起因であることはすぐに絞り込めた。

次に行ったのはブランチ間の差分確認だ。

```bash
# リリース前のタグと現在のHEADを比較
git diff v2.14.0 HEAD -- src/modules/visit-schedule/visit-schedule.service.ts

# コンフリクト解消前のコミットと後のコミットを比較
git diff abc1234 def5678 -- src/modules/visit-schedule/visit-schedule.service.ts
```

`git diff` でコンフリクト解消のコミットを見ると、フィルタリング条件の行が `- ` として消えていることが一目でわかった。消えたコードは数行だったが、医療システムにとって「表示すべきでないデータが見えてしまう」は重大な問題だ。

ロールバックかホットフィックスかの判断は5分で決めた。消えたコードの内容が明確で修正量が少なかったため、ホットフィックスブランチを切って修正・即日リリースを選択した。ロールバックは他の機能追加も巻き戻すため、影響範囲が広くなる懸念があった。

```bash
# ホットフィックスブランチを本番相当のタグから切る
git checkout -b hotfix/visit-schedule-filter v2.14.0
git cherry-pick <コンフリクト解消前のコミット>

# 消えた行を手動で復元し、コミット
git add .
git commit -m "fix: 来院スケジュール取得クエリの有効フラグフィルタを復元"
```

修正から本番適用まで約2時間で完了した。

## なぜコンフリクト解消ミスが起きやすいか

コンフリクト解消は、コードの内容と文脈を両方理解しながら「どちらのコードが正しく、どう組み合わせるか」を判断する作業だ。

今回の問題の根本は、**コンフリクトの解消者がリファクタリング側のチケット内容を十分に把握していなかった**ことにある。自分が書いたのは絞り込み条件追加のほうで、リファクタリング側は別のメンバーが進めていた。コンフリクトマーカーを機械的に処理するには、両方の意図を理解している必要がある。

また、コンフリクトが多い状態でのマージは集中力が切れやすい。「あと3箇所」という状況でミスが起きやすいのは、自分の経験からも確かだ。

## 再発防止策として導入したこと

### 1. コンフリクト解消PRのレビュープロセスを強化した

コンフリクト解消を含むコミットには、**変更されたファイルの担当者が必ずレビュアーに入る**ルールを追加した。自分だけで解消を完結させず、コンフリクトの相手側のコードを書いた人間が最終確認する。

### 2. コンフリクトが多いときはブランチを細かく分ける

「このPRは大きすぎる」と気づいたら、一度立ち止まってブランチを分割する習慣をつけた。同じファイルを同時期に複数人で触る状況を減らすのが最善策だ。

```bash
# 大きなリファクタリングPRを途中で分割するとき
git checkout feature/refactor-visit-schedule
git log --oneline main..HEAD  # 変更コミットを確認
# 分割の境界を決めてから、新しいブランチに必要なコミットだけ cherry-pick
```

### 3. 重要なコードパスのテストを追加した

今回問題になった「有効フラグによるフィルタリング」にはユニットテストが存在していなかった。本番データ相当の状態（無効化された被験者が存在する）でのテストケースを追加し、今後のリグレッションを検知できるようにした。

```typescript
describe('getVisitSchedules', () => {
  it('無効化された被験者の来院スケジュールは返さない', async () => {
    // active: false の被験者レコードを含むテストデータで検証
    const result = await service.getVisitSchedules();
    expect(result.every(s => s.subject.active === true)).toBe(true);
  });
});
```

## まとめ

コンフリクト解消は「マーカーを消す作業」ではなく、「2つのコードの意図を理解して統合する作業」だ。今回の経験で、コンフリクト解消コミットを**要注意レビューポイント**として扱う文化をチームに浸透させることができた。

治験システムのように「データの正確さが規制要件に直結するシステム」では、「ステージングで動いたから大丈夫」という判断基準だけでは不十分だ。テストデータと本番データの差異を意識したテストケースの設計が、こういった見落としを防ぐ最後の砦になる。

コンフリクト解消ミスはどのチームでも起きうる。大切なのは、起きたあとに原因と構造を正確に理解して、プロセスを変えることだと思っている。
