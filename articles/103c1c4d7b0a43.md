Next.jsを本番環境に乗せるとき、選択肢の多さで迷う。Vercel・AWS・Heroku…。どれを選ぶかはそのまま運用コストと性能に直結するため、入門段階では判断がつきにくい。実務で複数パターン経験した中で整理した基準をまとめる。

## レンダリング方式を先に把握する

| 方式 | 概要 | デプロイ制約 |
|---|---|---|
| SSG（静的生成） | ビルド時にHTMLを生成 | 静的ホスティングでOK |
| SSR | リクエスト時にサーバーで生成 | Nodeサーバーが必要 |
| ISR | 静的 + 一定時間で再生成 | Vercel or Nodeサーバー推奨 |
| App Router（RSC） | Server Components | Nodeサーバーが必要 |

## Vercel

GitHubと連携するだけでCI/CDが完成。SSG・SSR・ISR・App Routerすべてに対応。プレビューデプロイが自動で作られるのが開発体験として最高。ただしトラフィックが増えるとコストが跳ね上がる。

**向いているケース:** スタートアップ初期 / 個人プロジェクト / インフラに手をかけたくないチーム

## AWS S3 + CloudFront

SSGで書き出した静的ファイルをS3に置き、CloudFrontで配信する構成。コストが安く、AWSの他サービスとの連携が楽。SSR・ISR・RSCには対応できない。

当ブログ（taka-techblog.com）はAstro SSG + この構成で動いている。

**向いているケース:** SSGのみのブログ・LP / AWSに統一したいチーム

## AWS Amplify

AWSが提供するフルマネージドのホスティング + CI/CD。VercelのAWS版に近い。SSR対応（LambdaでNext.jsサーバーを動かす）。AWSに統一したい + SSRが必要なケースに向いている。

## Heroku

2026年現在、新規プロジェクトでHerokuを選ぶ理由は薄くなっている。コスパ面でもDX面でもVercelかFly.ioが上回っている。

## 選択フローチャート

```
SSGのみ？
├─ Yes → AWS S3 + CloudFront
└─ No（SSR/ISR/RSC使う）
    ├─ AWSに統一したい → AWS Amplify
    ├─ インフラ管理したくない → Vercel
    └─ 自前管理OK → EC2 + ALB または ECS
```

## 私の実際の選択

| プロジェクト | 選択 | 理由 |
|---|---|---|
| 当ブログ（Astro SSG） | AWS S3 + CloudFront | SSGのみ、コスト最安 |
| 社内Webアプリ（SSR） | AWS EC2 + ALB | 既存AWSインフラに統一 |
| 個人の検証用Next.js | Vercel | 設定ゼロで動く |
