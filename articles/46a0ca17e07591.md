入社して最初に言われた言葉が「このシステム、ベンダーに月40万払ってるんだけど何とかならない？」だった。

それから1年後、ベンダー委託費はゼロになり、年間500万円の削減を達成した。その全記録を書いておく。

## なぜ内製化を決断したか

当時の状況はこうだ。治験CRMのシステム一式を外部ベンダーに委託しており、月40万（年間480万）の固定費がかかっていた。問題はコストだけではない。

- **スピード**: 仕様変更の依頼から実装まで平均2〜3週間かかる
- **品質**: 仕様書を介したコミュニケーションにより、意図と実装がズレることが頻発
- **透明性**: コードが完全なブラックボックスで、障害対応時に身動きが取れない

「このまま外注し続けても、プロダクトは自分たちのものにならない」という危機感が積み上がった結果、内製化に踏み切ることになった。

## コード引き継ぎで直面した技術的負債

ベンダーからコードを受け取ったとき、正直に言うと絶望した。

- テストコードがほぼゼロ
- 環境構築手順書なし（DockerfileもCI設定もない）
- 命名規則がバラバラ、コメントも皆無
- ロジックがコントローラーに集中した典型的なFat Controller

まずやったのはコードリーディングからドキュメント化だ。動いているシステムを壊さずに理解するため、ひたすら読んでノートに書き続けた。この期間が2ヶ月ほどかかった。

## 1年で内製化を完遂するためにやったこと

### 技術移管フェーズ（1〜3ヶ月目）

既存コードを動かしながら、並行してローカル開発環境をDockerで整備した。

```bash
# まずは動く環境を作ることが最優先
docker-compose up -d
# テストが通ることを一つひとつ確認しながら進める
```

CI/CDはGitLab CI/CDで構築し直し、mainブランチへのマージで自動デプロイが走るフローを確立した。

### リファクタリングフェーズ（4〜8ヶ月目）

機能を壊さないようにしながら、少しずつリファクタリングを進めた。NestJSへの移行もこの時期に判断し、モジュール設計を一から引き直した。

テストを書きながら進めることで「壊していないか」の確信を持てるようになり、開発速度が格段に上がった。

### チーム構築（並行して）

未経験エンジニア2名を採用し、コーチング型で育てた。コードレビューを通じた学習機会を意図的に設計し、ドキュメントの整備も一緒に進めた。

## 結果：何が変わったか

- **コスト**: 月40万のベンダー委託費 → 0円（年間480万削減）
- **改修スピード**: 仕様確定から実装まで2〜3週間 → 即日〜3日
- **障害対応**: 「ベンダーに連絡して折り返し待ち」 → 自分たちでログを見て即対応

数字にならない変化もある。「このシステムは自分たちのものだ」というオーナーシップが生まれたことで、エンジニアチームの自律性が格段に上がった。

## 内製化して初めてわかった「外注の本当のコスト」

お金だけじゃない。本当のコストはスピードと学習機会の喪失だ。

外注している間、社内にノウハウが蓄積されない。仕様書を書く手間がかかる。認識のズレで手戻りが発生する。これらは全部、会社の競争力を削るコストだ。

小さなスタートアップほど、コアシステムは内製すべきだと今は確信している。

## まとめ

内製化は「コスト削減」ではなく「プロダクトの主権を取り戻す」行為だと思っている。

技術的負債まみれのコードを引き継いで、泥臭く1年かけて積み上げた結果、年間500万削減という数字が出た。でもそれ以上に、「自分たちで作れる」という自信がチームに根付いたことが一番の成果だった。

内製化を迷っている担当者の方がいれば、ぜひ背中を押したい。最初の半年は本当につらいけど、その先の景色は全然違う。
