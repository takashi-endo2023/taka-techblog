Next.jsプロジェクトを始めるとき、必ず直面するのがCSS設計の選択だ。styled-componentsかCSS Modulesか。この選択はプロジェクト全体のDXと保守性に影響するにもかかわらず、「なんとなく」で決めているケースが多い。

## styled-components とは

```tsx
import styled from 'styled-components';

const Button = styled.button<{ $primary?: boolean }>`
  padding: 0.5rem 1rem;
  background: ${(props) => (props.$primary ? '#7c6af7' : 'transparent')};
  border: 2px solid #7c6af7;
  border-radius: 6px;
  color: ${(props) => (props.$primary ? '#fff' : '#7c6af7')};
`;
```

**メリット:** PropsでスタイルをJSで動的に切り替えられる / コンポーネントとスタイルが1ファイル
**デメリット:** ランタイムでCSSを生成するためSSRが重い / App Routerとの相性が悪い

## CSS Modules とは

```tsx
// Button.module.css
.button { padding: 0.5rem 1rem; border: 2px solid #7c6af7; }
.primary { background: #7c6af7; color: #fff; }

// Button.tsx
import styles from './Button.module.css';
<button className={`${styles.button} ${styles.primary}`}>送信</button>
```

**メリット:** ビルド時に静的なCSSを生成 / App Router完全対応 / 学習コストが低い
**デメリット:** 動的スタイルの切り替えが冗長になることがある

## 比較表

| 観点 | styled-components | CSS Modules |
|---|---|---|
| 動的スタイル | ◎ | △ |
| パフォーマンス（SSR） | △ | ◎ |
| App Router対応 | △ | ◎ |
| 学習コスト | やや高い | 低い |
| バンドルサイズ | やや大きい | 小さい |

## 2026年時点の現実

正直なところ、**新規プロジェクトでstyled-componentsを選ぶ理由は薄くなっている**。

Next.js App RouterはServer Componentsを前提としており、ランタイムでCSSを生成するstyled-componentsはServer Component内では動作しない。CSS Modules、vanilla-extract、Tailwind CSSなどのゼロランタイムアプローチが主流になってきている。

## 選択基準まとめ

- **既存プロジェクト（Pages Router）でstyled-componentsが使われている** → そのまま継続
- **新規プロジェクトでApp Router** → CSS ModulesまたはTailwind CSS
- **動的テーマ切り替えが重要** → CSS変数 + CSS Modulesで対応できることが多い
