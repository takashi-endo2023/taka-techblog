REST APIの書き込みは読み取りより難しい。重複リクエスト・並行更新・トランザクションへの対応が必要になる。

## HTTPメソッドのべき等性

| メソッド | べき等 | 用途 |
|---|---|---|
| GET / HEAD | ✅ | 取得 |
| PUT | ✅ | 全体更新・作成 |
| DELETE | ✅ | 削除 |
| POST / PATCH | ❌ | 作成・部分更新 |

**べき等** = 同じリクエストを何度送っても結果が変わらない。べき等なAPIはリトライが安全。

## POSTの冪等性を確保する（Idempotency Key）

```ts
// クライアント：一意なキーを付与してリクエスト
await fetch('/api/orders', {
  method: 'POST',
  headers: { 'Idempotency-Key': crypto.randomUUID() },
  body: JSON.stringify({ items }),
});
```

```ts
// サーバ：同じキーのリクエストは前回の結果を返す
async createOrder(dto: CreateOrderDto, idempotencyKey: string) {
  const cached = await this.cache.get(`idempotency:${idempotencyKey}`);
  if (cached) return JSON.parse(cached); // 重複リクエスト → キャッシュを返す

  const order = await this.ordersRepository.create(dto);
  await this.cache.set(`idempotency:${idempotencyKey}`, JSON.stringify(order), 86400);
  return order;
}
```

## 楽観的ロック（ETag + If-Match）

複数ユーザーが同じリソースを同時編集するとき、後からの更新が前の変更を上書きする問題（Lost Update）を防ぐ。

```http
# 1. 取得時にETagを受け取る
GET /articles/1
→ ETag: "version-5"

# 2. 更新時にIf-Matchで指定
PUT /articles/1
If-Match: "version-5"

# バージョン一致 → 200 OK
# バージョン不一致 → 409 Conflict（他ユーザーが更新済み）
```

```ts
// NestJS実装
async update(id: number, dto: UpdateArticleDto, etag: string) {
  const article = await this.articlesRepository.findOne(id);
  if (`version-${article.version}` !== etag) {
    throw new ConflictException('他のユーザーによって更新されました');
  }
  return this.articlesRepository.update(id, { ...dto, version: article.version + 1 });
}
```

## 悲観的ロック（SELECT FOR UPDATE）

競合が頻繁・競合コストが高い場合（在庫管理・決済など）に使う。

```ts
return this.dataSource.transaction(async (manager) => {
  const product = await manager.findOne(Product, {
    where: { id: productId },
    lock: { mode: 'pessimistic_write' }, // SELECT FOR UPDATE
  });
  if (product.stock < quantity) throw new BadRequestException('在庫不足');
  product.stock -= quantity;
  return manager.save(product);
});
```

## 楽観的 vs 悲観的ロックの使い分け

| 観点 | 楽観的 | 悲観的 |
|---|---|---|
| 前提 | 競合は稀 | 競合が頻繁 |
| パフォーマンス | 高い | 低い（待ちあり） |
| 向いているケース | CMS・ドキュメント編集 | 在庫・決済 |
