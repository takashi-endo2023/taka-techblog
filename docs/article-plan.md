### 🤖 AI・開発効率化（優先度: 高）

#### 1. AI×業務フロー効率化：社内コンサル的なアプローチ
- **想定ファイル**: `ai-business-process-consulting.md`
- **概要**: エンジニアがAIを社内に広める立場になったとき。技術者だけの話ではなく、非エンジニアの業務（議事録・報告書・メール下書き・データ整理）にどうAIを届けるか。社内ワークショップ・ハンズオンの設計
- **自分の切り口**: 医療系で「個人情報をAIに入力していいか？」という懸念への向き合い方

#### 2. 結合試験をボタン1つで完結させる：CI自動化の設計
- **想定ファイル**: `integration-test-automation.md`
- **概要**: 手動で行っていた結合試験フローをCI/CDに組み込む。テストシナリオの設計・Playwright等E2Eツール選定・ステージング環境での自動実行・レポート生成まで
- **自分の切り口**: 治験システムでの「証跡が残る自動テスト」。規制産業では「テストが実行された証拠」そのものが求められる
- **関連記事**: `frontend-testing-practical-approach.mdx`, `github-actions-oidc-aws.md`

#### 3. PRマージ時の品質ゲート：マージ前に問題を自動検知する仕組み
- **想定ファイル**: `pr-quality-gate-automation.md`
- **概要**: GitHub ActionsでPRマージ前に自動でチェックする仕組み。型チェック・lint・テスト・セキュリティスキャン（dependabot/trivy）・カバレッジしきい値。Branch Protectionルールとの連携
- **自分の切り口**: 少数精鋭チームでレビュワーが1人しかいないとき、機械的なチェックに頼ることで人間のレビューを設計判断に集中させた体験

#### 4. Claude Code × チーム開発：1人で使うツールからチームへ広げるとき
- **想定ファイル**: `claude-code-team-adoption.md`
- **概要**: Claude Codeを自分だけが使っている状態から、チームに展開するときの課題。コスト・セキュリティポリシー・プロンプト共有・使いどころのすり合わせ
- **自分の切り口**: 規制産業でのAIツール導入承認プロセス。「このデータをAIに渡していいか」の判断基準を作った話

#### 5. AIを使ったコードレビュー支援：人間のレビューと何が違うか
- **想定ファイル**: `ai-code-review-assistant.md`
- **概要**: AIに初回レビューをさせてから人間がレビューする2段階方式。AIが得意なこと（命名・重複・バグパターン）と人間が必要なこと（設計判断・仕様理解）の棲み分け
- **関連記事**: `code-review-culture.md`

### 🏥 医療IT・治験・規制対応（優先度: 高）

#### 6. CSVドキュメントを1から作った話——バリデーションプロトコルの現実
- **想定ファイル**: `csv-documentation-reality.md`
- **概要**: csv-validation-guide.mdは「CSVとは何か」の解説。こちらは「実際に作ると何が大変か」の実体験。URS・FRS・IQ/OQ/PQ・トレーサビリティマトリクスを1から書いたときの工数・ハマりポイント・Excelで管理する限界
- **自分の切り口**: 「誰もやり方を教えてくれない」状態でGALP/GAMP5を読みながら手探りで作った話
- **注意**: csv-validation-guide.mdとは別の切り口（作業実体験）にする

#### 7. PPD・CROの品質監査に技術者として対応した話
- **想定ファイル**: `cro-audit-tech-response.md`
- **概要**: 製薬会社の委託先（CRO/PPDなど）が実施する品質監査でシステム担当として同席したときに何を聞かれるか。監査員の視点・よく指摘される項目・エビデンスとして準備すべき資料
- **自分の切り口**: system-audit-experience.mdとの違いは「製薬会社側から来た監査」vs「自社への監査対応」。具体的に聞かれた質問・詰められたポイント
- **注意**: system-audit-experience.mdはスポンサー監査の話。これはCRO・PPD等のグローバル品質組織からの監査

#### 8. 製薬会社からのシステム解凍——変更管理と「凍結」の現実
- **想定ファイル**: `system-freeze-change-control.md`
- **概要**: 治験稼働中はシステムを「凍結」し、変更には製薬会社の承認が必要になる。このプロセス（Change Request → Impact Assessment → Validation → Sponsor Approval → 解凍）の実体験。機能追加1つに3ヶ月かかったときの話
- **自分の切り口**: エンジニアの感覚では「バグ修正なのに3ヶ月待ち」の理不尽さと、規制の合理性の両方を理解した経験。スピードと安全のトレードオフを一番感じた場所

#### 9. 21 CFR Part 11 と電子署名——医療ITエンジニアが知るべき規制の最低限
- **想定ファイル**: `21cfr-part11-tech-guide.md`
- **概要**: FDA規制21 CFR Part 11が求める電子記録・電子署名の要件をエンジニア目線で整理。監査証跡・タイムスタンプ・署名者の一意識別・改ざん防止の実装レベルでの解釈
- **自分の切り口**: 「これを実装したらPart 11準拠か？」を製薬会社に確認するまでの不確実性。エンジニアには法的判断ができない部分をどう扱うか
